# /onboard-check — 受注直後チェックリスト

受注が決まった直後、迷わず動けるための定型チェックリスト。
ユーザーからは案件タイプ（LP / Figma実装 / 既存改善）と、分かっている範囲で案件情報を1〜2行で受ける。

## 出力順（固定）

### 1. 先方に最初に送る確認メッセージ（コピペ用）

案件タイプに応じてテンプレを差し替える。必ず次の要素を含める:
- 受注御礼＋開始目安日
- 不明点を確認したい旨
- 下記5項目を1メッセージにまとめて質問

**共通5質問:**
1. 最終納期・希望公開日
2. 納品形式（Figmaリンク / GitHub / ZIP / Vercel公開URL など）
3. 参考にしたいサイト・トンマナ・NGイメージ
4. 支給素材（テキスト・画像・ロゴ・ブランドガイド）の有無と入手経路
5. コミュニケーション手段と応答時間帯

### 2. 先に決めるデザイン方針（社内メモ用）

- ターゲット/1画面1感情のコアメッセージ
- 界隈トーン（地雷／天使／水色／その他）
- カラートークン（主軸＋補助3色）
- フォントスタック（SIL OFL商用セーフ）
- 参考LPを2〜3本ピックアップ

### 3. 技術スタック決定

| 案件タイプ | 推奨スタック |
|---|---|
| 静的LP | HTML / CSS / Vanilla JS / Vercel |
| Figma→実装 | 同上、必要に応じReact |
| 既存改善 | 既存スタック維持・最小差分 |

### 4. 作業リポ/ブランチ戦略

- 新規repo: `gh repo create <name> --private` （命名: `client-<案件名>-lp`）
- 既存repo改修: feature branch `claude/<issue-slug>` を切る
- main直push禁止・PR経由

### 5. 着手順（最小1日で出せる形）

1. Figmaで1画面だけ下書き（Hero + CTA）
2. HTML/CSS雛形（セクション枠だけ）
3. Figma完成 → 残りセクション実装
4. レスポンシブ調整
5. Lighthouseで最低スコア確認（Performance/Accessibility 80以上）
6. Vercel preview URL を先方に共有
7. 修正ループ → OK後 production デプロイ

### 6. 納品前の最終チェック（/delivery-check に接続）

`/delivery-check` を呼ぶ。重複するので本コマンドでは出さない。

### 7. 受注直後にやらないこと

- 先方の希望を聞く前にデザインを始めない
- 勝手に追加機能を提案しない（スコープ逸脱）
- 納期見込みを曖昧に回答しない（必ず具体日で返す）

## 使い方

```
/onboard-check LP制作 / CrowdWorks / 納期1週間 / 美容系
```

→ 上記6ブロックを案件タイプに合わせて埋めて出す。
